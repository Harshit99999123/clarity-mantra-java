# Clarity Mantra Backend API

Frontend integration contract for the React client.

Base URL:

```text
http://localhost:8080/api/v1
```

Swagger:

```text
/api/v1/swagger-ui.html
```

OpenAPI JSON:

```text
/api/v1/v3/api-docs
```

## Auth model

- Protected endpoints require `Authorization: Bearer <accessToken>`
- JWT is returned from:
  - `POST /auth/register`
  - `POST /auth/login`
  - `POST /auth/google`
- The backend returns numeric SQL IDs, not UUIDs

## Standard response envelope

All non-streaming success responses:

```json
{
  "success": true,
  "message": "Human readable status",
  "data": {},
  "timestamp": "2026-03-20T00:00:00Z",
  "correlationId": "request-correlation-id"
}
```

Typical error shape from the application:

```json
{
  "success": false,
  "message": "Bad Request",
  "errors": [
    {
      "code": "VALIDATION_ERROR",
      "detail": "email: must be a well-formed email address"
    }
  ],
  "timestamp": "2026-03-20T00:00:00Z",
  "correlationId": "request-correlation-id"
}
```

Some framework-level validation errors may come back as Spring Problem Details:

```json
{
  "type": "about:blank",
  "title": "Bad Request",
  "status": 400,
  "detail": "Invalid request content.",
  "instance": "/api/v1/auth/google"
}
```

## Public endpoints

### `GET /health`

Basic app health.

Response:

```json
{
  "success": true,
  "message": "Service is healthy.",
  "data": {
    "status": "ok",
    "service": "clarity-mantra-core"
  },
  "timestamp": "2026-03-20T00:00:00Z",
  "correlationId": "request-correlation-id"
}
```

### `POST /auth/register`

Create an email/password user and immediately receive a JWT.

Request:

```json
{
  "fullName": "Aarav Sharma",
  "email": "aarav@example.com",
  "password": "StrongPass123",
  "preferredLanguage": "en"
}
```

Response:

```json
{
  "success": true,
  "message": "User registered successfully.",
  "data": {
    "accessToken": "jwt-token",
    "expiresInSeconds": 86400,
    "user": {
      "id": 12,
      "fullName": "Aarav Sharma",
      "email": "aarav@example.com",
      "preferredLanguage": "en",
      "authProvider": "EMAIL"
    }
  }
}
```

### `POST /auth/login`

Login with email/password.

Request:

```json
{
  "email": "aarav@example.com",
  "password": "StrongPass123"
}
```

Response shape is the same as register.

### `POST /auth/google`

Login/signup using a Google ID token generated by the client.

Request:

```json
{
  "idToken": "google-id-token-from-client",
  "preferredLanguage": "en"
}
```

Response shape is the same as register.

Notes:

- The backend validates the Google token server-side
- `idToken` must be a real Google-issued ID token for the configured client ID

### `POST /conversations/{conversationId}/messages`

Send one user message and receive one mentor response.

Request:

```json
{
  "message": "I feel confused about my next career step and want grounded guidance.",
  "inputMode": "TEXT"
}
```

Response:

```json
{
  "success": true,
  "message": "Message posted successfully.",
  "data": {
    "id": 102,
    "role": "MENTOR",
    "inputMode": "TEXT",
    "content": "Long mentor reply...",
    "reflectionQuestion": "What feels most important to resolve first?",
    "references": [
      {
        "reference": "2.47",
        "translation": "You have a right to perform your prescribed duty...",
        "themes": [
          "duty",
          "detachment"
        ]
      }
    ],
    "createdAt": "2026-03-20T00:01:03Z"
  }
}
```

Behavior:

- The backend stores the user message
- It calls the AI service
- It stores and returns the mentor message
- The conversation summary/title may be updated in the background persistence step

### `POST /conversations/{conversationId}/messages/stream`

Stream the mentor response with Server-Sent Events.

Request:

```json
{
  "message": "Help me reflect slowly on what I should prioritize next.",
  "inputMode": "TEXT"
}
```

Response type:

```text
text/event-stream
```

Observed event sequence:

```text
event: meta
data: {"references":[{"source":"Bhagavad Gita As It Is","reference":"11.22"}]}

event: token
data: {"text":"I"}

event: token
data: {"text":" sense"}

event: token
data: {"text":" your"}

event: done
data: {}
```

Frontend notes:

- Use `fetch` streaming or an SSE-compatible parser for POST streaming
- The backend persists the final mentor message after streaming completes
- After stream completion, call `GET /conversations/{conversationId}` if you want the final stored transcript

{

  "data": {

### `GET /conversations/{conversationId}/insight-card`

Fetch the latest generated insight card for the conversation.

## Feedback

### `POST /feedback`

Submit general or session feedback.

Request:

```json
{
  "conversationId": 21,
  "type": "SESSION",
  "rating": 5,
  "comment": "Helpful and grounded."
}
```

`conversationId` can be `null` for non-session feedback.

Response:

```json
{
  "success": true,
  "message": "Feedback submitted successfully.",
  "data": {
    "id": 3,
    "type": "SESSION",
    "rating": 5,
    "comment": "Helpful and grounded.",
    "conversationId": 21,
    "createdAt": "2026-03-20T00:08:00Z"
  }
}
```

## Frontend implementation notes

- Store `accessToken` after auth and send it as a bearer token on every protected request
- All IDs are numeric
- Use `GET /auth/me` after app boot if you want to restore the logged-in user from stored JWT
- Use `GET /conversations/{id}` after posting or streaming if the UI needs the fully refreshed transcript
- `preferredLanguage` is a plain string like `en` or `hi`
- The backend currently supports the `TEXT` input mode in the verified flow
